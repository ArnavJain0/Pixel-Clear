─────────────────────────────────────────────────────────────
1. WHAT THE PROJECT DOES
─────────────────────────────────────────────────────────────
Pixel-Clear is a web application that lets a user upload a blurry
or low-quality photograph and receive a 4× super-resolved, enhanced
version in return. The enhancement pipeline is:

  1. User uploads image via the browser.
  2. The browser resizes the image to ≤ 480 px on the longest side
     before sending it (no heavy work on the server during upload).
  3. The server saves the upload to /tmp and launches a short-lived
     Python subprocess (worker.py).
  4. worker.py loads the RealESRGAN × 4 model (OpenVINO format),
     runs a single forward pass, applies premium post-processing
     filters, saves the result JPEG, and then EXITS.
  5. The browser polls /status/<task_id> every second and shows the
     progress bar.
  6. When the task is complete the result is displayed in a
     before/after comparison slider.

3. FILE-BY-FILE EXPLANATION
─────────────────────────────────────────────────────────────

── server.py ──────────────────────────────────────────────
Role      : Permanent web process.
Framework : FastAPI + uvicorn.
RAM usage : ~30–40 MB at idle.

Key design decisions
  • Imports NOTHING heavy (no cv2, no numpy, no openvino).
    All those are imported only inside worker.py.
  • INFER_SEMAPHORE = threading.Semaphore(1)
    Only ONE subprocess may run at a time.  This is the primary
    guard against RAM spikes from concurrent requests.
  • Uses SQLite with WAL journal mode for safe concurrent reads/writes.
  • A watchdog thread deletes result files older than 1 hour every
    5 minutes to prevent disk fill.
  • All routes return JSON with integer progress (never null).

Routes
  GET  /                    → serves index.html
  POST /upload              → receives JPEG blob, starts worker thread
  GET  /status/<task_id>    → returns {status, progress, remaining, message}
  GET  /image/<task_id>     → returns result JPEG for display
  GET  /download/<task_id>  → returns result JPEG as attachment

── worker.py ──────────────────────────────────────────────
Role     : Short-lived inference subprocess.
Lifetime : Spawned per request, exits when done.

Execution flow
  1. Read image from /tmp/<task_id>_in.jpg
  2. Resize to ≤ 480 px (safety cap)
  3. Load weights/model.xml with OpenVINO (LATENCY hint, 2 CPU threads)
  4. Normalise → CHW → float32 → single forward pass
  5. Denormalise → apply CLAHE + unsharp-mask + brightness boost
  6. Save to static/results/<task_id>.jpg at JPEG quality 85
  7. Exit — OS reclaims ALL memory

RAM usage : peaks at ~280–350 MB during inference, then drops to 0.

── converter.py ───────────────────────────────────────────
Role : One-time local conversion tool.
Run  : python converter.py   (requires torch + torchvision)
       Do NOT run this on Render — it is for local developer use only.

What it does
  • Downloads RealESRGAN_x4plus_anime_6B.pth (compact 6-block variant)
    from GitHub if not present.
  • Loads the weights into the RRDBNet architecture (model_arch.py).
  • Converts to OpenVINO IR format (fp16) using ov.convert_model().
  • Saves weights/model.xml + weights/model.bin.
  • The resulting files are committed to Git so Render never needs
    to run the converter.

── templates/index.html ───────────────────────────────────
Framework : Tailwind CSS (CDN), Vanilla JS, Google Fonts (Outfit).

Key JavaScript functions
  resizeImage(file, maxDim=480)
      Uses an off-screen HTML5 Canvas to downscale the image to
      ≤ 480 px before it is sent.  This offloads the costliest
      decoding step from the server to the user's browser.

  handleUpload(file)
      Calls resizeImage, shows the original as the "before" preview,
      then POSTs the optimised blob to /upload.

  pollStatus(taskId)
      Fetches /status/<taskId> every 1 s.
      Guards: typeof data.progress === 'number' — prevents the
      "undefined%" bug when the task is still in 'starting' state.

  showResult(taskId)
      Displays the before/after comparison slider and download link.

Snake mini-game
      A playable Snake game is shown while the image is processing
      so the user has something to do.  It runs fully in-browser and
      uses no server resources.

── Procfile ────────────────────────────────────────────────
Contents:
  web: uvicorn server:app --host 0.0.0.0 --port $PORT --workers 1

Why uvicorn and not gunicorn?
  Gunicorn with multiple workers would fork the process, multiplying
  the base RAM.  A single uvicorn worker is correct for this app
  because all concurrency is handled inside Python threads and the
  semaphore.

── requirements.txt ─────────────────────────────────────────
fastapi
uvicorn[standard]
python-multipart
openvino>=2025.0.0
opencv-python-headless
numpy>=1.26.4

Note: Flask, flask-cors, flask-compress, requests, gunicorn are
intentionally REMOVED.  They are not needed and add startup RAM.
openvino and opencv are listed because worker.py needs them when
it is spawned as a subprocess inside the same Python environment.

─────────────────────────────────────────────────────────────
4. RAM BUDGET ON RENDER FREE TIER (512 MB)
─────────────────────────────────────────────────────────────
Phase                         | Approx RAM
------------------------------|-------------
Idle (server only)            | ~35–45 MB
Receiving upload              | +5 MB (tiny blob)
Worker subprocess running     | +280–350 MB (peak, then drops to 0)
Concurrent uploads (queued)   | still one worker at a time → safe
------------------------------|-------------
Worst case                    | ~395 MB  (well under 512 MB)

No global model cache is kept in server.py.  Every time a user
sends a photo, a fresh worker.py is started and exits.
Python's RAM is NOT reliably returned via gc.collect(), but OS
process exit IS guaranteed to free everything.

─────────────────────────────────────────────────────────────
5. THE ROOT CAUSE OF ALL PREVIOUS OOM CRASHES
─────────────────────────────────────────────────────────────
Until this fix the Procfile pointed at app:app (the old Flask file).
server.py existed but was NEVER started.

app.py imported openvino at the top level, so the model was
compiled into memory from startup and kept alive for the entire
lifetime of the process (≈ 200–300 MB always resident).
Every image request ADDED another 200–300 MB on top of that.
Total = 400–600 MB → OOM guaranteed.

Fix summary
  • Procfile now points at server:app (FastAPI)
  • app.py deleted — can no longer be accidentally loaded
  • server.py imports zero heavy libraries
  • All heavy work is in worker.py (exits → frees RAM)
  • Browser resizes before upload → server never sees huge data

6. HOW TO DEPLOY (step-by-step)
─────────────────────────────────────────────────────────────
Prerequisites
  • Git repository connected to Render
  • model.xml and model.bin already in weights/ (generated locally
    by running `python converter.py` once)

Steps

  1. Make sure app.py does NOT exist in the repo:
       git rm app.py   (if it is tracked)

  2. Commit all changed files:
       git add server.py worker.py Procfile requirements.txt \
               templates/index.html
       git commit -m "Final OOM fix: FastAPI + subprocess worker"
       git push origin main

  3. On Render, trigger a Manual Deploy (or it auto-deploys on push).

  4. Wait for the build to finish (it installs openvino which takes
     ~60–90 s).

  5. Open the live URL. Upload any photo. The progress bar should
     move from 0% → 5% → 15% → 100% in roughly 3–6 seconds on
     Render's free CPU.

  6. Check Render logs. You should see:
       INFO:     Application startup complete.
       [worker-thread] ...
       OK: saved static/results/<uuid>.jpg
     and NO "memory limit exceeded" messages.

7. HOW TO RUN LOCALLY
─────────────────────────────────────────────────────────────
  # Install deps (add torch only for converter)
  pip install -r requirements.txt

  # (First time only) convert the model
  pip install torch torchvision
  python converter.py

  # Start the server
  uvicorn server:app --host 0.0.0.0 --port 5000 --reload

  # Open http://localhost:5000

8. HOW TO TEST THE WORKER IN ISOLATION
─────────────────────────────────────────────────────────────
  python worker.py input.jpeg /tmp/out.jpg weights/model.xml

  On success you will see:
    OK: saved /tmp/out.jpg
  The file /tmp/out.jpg is the 4× enhanced result.

9. COMMON ERRORS AND FIXES
─────────────────────────────────────────────────────────────
Error                        | Cause                | Fix
-----------------------------|----------------------|------------------------
"model.xml not found"        | weights not in repo  | Run converter.py locally
                             |                      | and commit the files
"Unsupported file type"      | non-JPEG/PNG upload  | User must upload JPEG/PNG
"File too large"             | > 5 MB after resize  | Reduce JS maxDim or bump
                             |                      | MAX_UPLOAD_BYTES
Progress bar shows undefined | progress field null  | Fixed in index.html
Task stuck in "starting"     | worker crashed       | Check Render logs for
                             | silently             | worker stderr output
OOM restart from Render      | app.py still running | Delete app.py, re-deploy

10. MODEL DETAILS
─────────────────────────────────────────────────────────────
Architecture : RealESRGAN × 4 (anime_6B variant)
              6 RRDB blocks (compact) instead of the 23-block original
              Chosen for ~4× speed and ~60% lower RAM vs full model
Scale factor : 4× (a 480 × 360 input → 1920 × 1440 output)
Format       : OpenVINO IR (fp16 compressed)
Size         : model.xml ≈ 1.7 MB, model.bin ≈ 32 MB

Post-processing filters applied after the model:
  • CLAHE (Contrast Limited Adaptive Histogram Equalisation)
    Enhances local contrast without washing out flat regions
  • Unsharp-mask at σ = 2.0
    Adds perceived sharpness
  • Mild brightness/contrast boost (alpha=1.05, beta=2)
